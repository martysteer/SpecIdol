# SpecIdol Design Critique

**Date:** 2026-04-19
**Reviewer:** Claude (Impeccable Design Review)

## Anti-Patterns Verdict

**PASS** — This is decisively NOT AI slop.

This interface has a strong, intentional aesthetic: retro terminal/CRT game show. It commits fully to:
- Pure black (#000) + neon green (#0f0) + neon yellow (#ff0) palette
- Press Start 2P bitmap font (intentional retro choice, not SF Pro)
- CRT scanline effects (`repeating-linear-gradient` for audience view)
- Text shadows that glow (`text-shadow: 0 0 10px #0f0`)
- Zero gradients, zero glassmorphism, zero rounded corners

**This looks like it was designed for a speculative fiction convention game show, not generated by ChatGPT.** The aesthetic is campy, deliberate, and cohesive. It's the opposite of "neutral dark mode with purple accents."

---

## Overall Impression

**What works:** The commitment to the bit. This feels like you're using a terminal at a hacker convention in 1985. The retro game show aesthetic is **consistent** across all four views (index, controller, judge, audience). The typography is legible despite the bitmap font. The modal system you just added maintains the aesthetic instead of breaking it with generic Material Design alerts.

**What doesn't work:** Information density varies wildly between screens. The controller page is **cluttered** — you're asking users to scan a three-column session info grid, a collapsible two-column story management layout, round controls, and tiny session stats all at once. The judge view is the opposite: a giant red button and... almost nothing else. These don't feel like they belong to the same design system — one is dense, one is sparse.

**Biggest opportunity:** Unify the information hierarchy across views. The controller needs breathing room. The judge needs more context. Both could learn from the audience view, which balances information (timer, judge panels, story text) without feeling cramped.

---

## What's Working

1. **Aesthetic commitment**: You chose "retro terminal CRT game show" and every decision reinforces it. The scanlines on the audience view, the bitmap font, the glow effects — it's cohesive. You're not hedging. This is rare.

2. **Custom modal system**: The green-on-black modals with Press Start 2P copy maintain the aesthetic instead of breaking immersion with browser alerts. The confirmation prompts ("Is your opinion the truth?", "Do you like to watch?") are **playful and on-brand**. This is good microcopy.

3. **Session stats unification**: You just unified the `X stories | Y judges | Z audience` display across index.html and control.html. Same structure, same styling, same socket events. That's disciplined design — you're building a system, not duct-taping screens together.

## Priority Issues

### 1. Controller information overload
**What**: The controller page has seven distinct UI regions competing for attention: session code (huge), session stats (tiny), three danger buttons, collapsible story form, story queue, round controls, and selected story display. No clear hierarchy.

**Why it matters**: Controllers need to act fast during a live event. Right now their eyes have to scan the entire page to find "Start Round" or see which story is selected. Cognitive load kills performance under pressure.

**Fix**:
- Move session stats (stories/judges/audience) into the session code box — it's currently massive and wastes space. Shrink the code, stack the stats below it.
- Make the **selected story title** much more prominent. Right now `#selectedStory` is yellow text in the round controls — that should be **as big as the session code** because it's what's about to be performed.
- The three danger buttons (Eject/Shutdown/Delete) are red-on-dark-red. Make them **actually dangerous-looking**: bigger hit targets, more contrast, maybe even move them to a separate "Session Management" section that's collapsed by default.

**Command**: `/arrange` to restructure the controller layout for better hierarchy.

### 2. Judge view has no peripheral awareness
**What**: Judge view shows: story title, buzz button, timer. That's it. Judges can't see how many other judges are connected, how many have already buzzed, or any sense of the game state beyond their own button.

**Why it matters**: This is a **social performance**. Judges want to know if they're the first to buzz or the last holdout. That tension is part of the game. Right now they're isolated.

**Fix**:
- Add a subtle "X/Y judges connected" indicator (small, non-distracting).
- Show buzz status for other judges — maybe as small colored dots or a simple "2 buzzed / 3 total" counter.
- This info should be **present but not prominent** — the button is still the star.

**Command**: `/delight` to add contextual awareness that makes the game more social without cluttering the core interaction.

### 3. Session button layout breaks on many sessions
**What**: On index.html, session buttons are stacked vertically in a single column. If there are 5+ active sessions, you're scrolling.

**Why it matters**: At a convention with multiple concurrent sessions, the join page becomes a scrolling list. That's a poor use of screen space and slows down the "find and join" workflow.

**Fix**:
- Use a **grid layout** for session buttons (2 columns on desktop, 1 on mobile).
- Session buttons should be **equal height** regardless of stats text length.
- Consider adding a filter/search if this is meant to scale beyond 10 sessions.

**Command**: `/arrange` to restructure the session list for better space efficiency.

### 4. Story queue has no empty state guidance
**What**: When the controller first creates a session, the story queue is empty. It just says nothing — or shows an empty `<ul>`.

**Why it matters**: New users don't know what to do. Is the app broken? Where do stories come from? An empty state should **guide the next action**.

**Fix**:
- Show a message in the story queue when empty: "No stories yet. Add one to get started." with an arrow pointing to the "Add Story" form.
- Or: Auto-expand the "Add Story" disclosure if the queue is empty.
- This is basic onboarding.

**Command**: `/onboard` to add helpful empty states that guide users toward the next action.

### 5. Typography scale lacks breathing room
**What**: The bitmap font (Press Start 2P) is set at 0.6rem for stats, 0.9rem for buttons, 1.5rem for headings. These jumps are **too close together** — there's not enough differentiation between body text and headings.

**Why it matters**: Visual hierarchy depends on **contrast**. When all the text sizes are within a 2.5x range, nothing stands out. Compare this to a proper typographic scale (1.2x, 1.5x, 2x, 3x) where headings **feel** like headings.

**Fix**:
- Increase the heading scale: H1 should be **3rem+**, H2 should be **1.5-2rem**, body stays at 0.9-1rem.
- Session stats can stay tiny (0.6rem) — they're meant to be peripheral.
- The bitmap font is already hard to read at small sizes. Don't make it worse by using it at mid-sizes too.

**Command**: `/typeset` to establish a clearer typographic hierarchy with better size differentiation.

## Minor Observations

- **Audience CRT effect is subtle**: The scanlines are there but barely visible. If you're going for CRT, consider adding a slight flicker animation or vignette to sell it harder.
- **Danger buttons are hard to read**: Dark red (#800) background with red (#f00) text is low contrast. Hover state fixes this, but resting state is muddy.
- **Mobile responsiveness exists but isn't tested**: You have `@media (max-width: 768px)` but the controller page would be a nightmare on a phone — too many regions, too much density.
- **Disabled button state is subtle**: `opacity: 0.3` works, but bitmap fonts at low opacity are hard to read. Consider a different visual treatment (e.g., desaturate + reduce text shadow glow).

---

## Questions to Consider

- **What if the controller had a "live mode" vs "setup mode"?** Setup mode shows story management. Live mode hides it and focuses on round controls + selected story. Right now everything is visible at once.

- **What if judges could see a tiny preview of the story text?** Just the first line, so they know what they're about to judge. Or is the mystery intentional?

- **What if the audience view had sound?** You have `new Audio('tick.mp3')` in the code. Are you using sound? If not, why have the scaffolding? If yes, how do you communicate that to users (mute button)?

- **What would a "spectator mode" look like?** Right now you have Audience (projector) and Judge (participant). What if someone just wants to watch remotely without the full CRT projector aesthetic? Or is the CRT part of the show?

- **Does the bitmap font work at 0.6rem on a projector 20 feet away?** The audience view will be displayed on a big screen. Have you tested legibility at distance? Bitmap fonts get **muddy** when scaled.

---

**Final note:** This is a **designed** interface with a clear point of view. Most of the issues are about information architecture and hierarchy, not aesthetic flailing. Fix the density problems on the controller, give judges more context, and tighten the typography scale. The retro terminal aesthetic is working — don't water it down.
