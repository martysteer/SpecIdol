# Speculative Idol — Design Spec

A live "Pop Idol for writers" web app for a speculative fiction convention. Readers read stories aloud (with auto-scrolling text), three editor-judges buzz them out with big red buttons, and an audience watches on a projector. If the reader survives 2 minutes without all three judges buzzing, they win.

## Architecture

### Stack
- **Frontend**: Static HTML5/CSS/JS, no build step, no framework. One external font (Google Fonts CDN). Retro/campy aesthetic.
- **Backend**: Python WebSocket relay server (~100 lines). One pip dependency (`websockets`). Holds current round state in memory.
- **Hosting**: Static files served from Mythaxis web server at a path outside the Hugo site. Python WebSocket server runs alongside.

### File Structure

```
./www/
  index.html        — landing: enter story code or create session
  audience.html     — projector: scrolling text, timer, judge indicators, sound
  judge.html        — mobile: big red button, timer, story title, animated X
  control.html      — reader/controller: story queue, round controls, import/export
  style.css         — shared retro styles, pixel fonts, CRT effects
  app.js            — shared WebSocket client, message protocol, state management
  sounds/           — short audio files (.wav or inline base64)
    buzz.wav
    victory.wav
    defeat.wav
    tick.wav

./server/
  relay.py           — Python WebSocket relay with session state
  requirements.txt   — websockets
```

## Session Model

### Story Code
- 4 uppercase alphabetic characters (e.g. "XKRM")
- Generated by controller when creating a session
- Persists across all rounds in the event
- Judges join once with the code, stay connected through all stories

### Roles
- **Controller** (1): manages stories, controls rounds
- **Judge** (3+): buzzes stories. Judge count is a variable, not hardcoded to 3.
- **Audience** (any): read-only view of the action

### Joining
- `index.html` landing page: "Create Session" (becomes controller) or "Join Session" (enter code, pick role: judge or audience)
- Judges pick a judge number (1, 2, or 3) from available slots on join. Server rejects if slot taken.

## Server State

Python dict in memory. No database. Lost on server restart.

```python
session = {
    "code": "ABCD",
    "config": {
        "timer_duration": 120,    # seconds, global config
        "speed_options": [1, 2, 3],
        "judge_count": 3
    },
    "stories": [
        {"title": "The Last Starship", "text": "It was a dark..."}
    ],
    "current_round": {
        "story_index": 0,
        "title": "The Last Starship",
        "text": "It was a dark...",
        "start_time": null,
        "speed": 1,
        "paused": false,
        "pause_time": null,
        "buzzes": [],
        "status": "waiting"  # waiting | running | paused | defeat | victory
    },
    "history": [
        {
            "title": "Some Story",
            "buzzes": [
                {"judge_id": 1, "time": 42.3},
                {"judge_id": 2, "time": 75.1},
                {"judge_id": 3, "time": 98.7}
            ],
            "outcome": "defeat",
            "duration": 98.7
        }
    ]
}
```

## WebSocket Message Protocol

All messages are JSON. Format: `{"type": "...", "data": {...}}`

### Client → Server
| Type | Data | Sender |
|------|------|--------|
| `join` | `{code, role, judge_id?}` | any |
| `round_start` | `{story_index}` | controller |
| `speed_change` | `{speed: 1\|2\|3}` | controller |
| `pause` | `{}` | controller |
| `resume` | `{}` | controller |
| `buzz` | `{judge_id}` | judge |
| `reset_round` | `{}` | controller |
| `add_story` | `{title, text}` | controller |
| `remove_story` | `{story_index}` | controller |
| `import_session` | `{stories: [...], config: {...}}` | controller |

### Server → Client
| Type | Data | Recipients |
|------|------|------------|
| `session_state` | full session dict | joining client |
| `round_started` | `{title, text, start_time}` | all |
| `speed_changed` | `{speed, timestamp}` | all |
| `paused` | `{timestamp, elapsed}` | all |
| `resumed` | `{timestamp, elapsed}` | all |
| `buzzed` | `{judge_id, time}` | all |
| `round_ended` | `{outcome, buzzes, duration}` | all |
| `round_reset` | `{}` | all |
| `story_added` | `{index, title}` | all |
| `story_removed` | `{index}` | all |
| `session_imported` | `{story_count}` | all |
| `error` | `{message}` | sender |

## The Four Views

### 1. Landing Page (`index.html`)

- Big retro "SPECULATIVE IDOL" title in pixel font
- Two buttons: "Create Session" / "Join Session"
- Join flow: text input for 4-char story code, role selector (Judge / Audience), judge number picker (if judge)
- Create flow: generates code, redirects to `control.html?code=ABCD`
- Join redirects to `judge.html?code=ABCD&judge=1` or `audience.html?code=ABCD`

### 2. Audience/Projector View (`audience.html`)

**Layout**: optimized for 16:9 landscape projection.

- **Header**: story title, centered
- **Body**: auto-scrolling story text, large readable font (monospace or clean sans-serif)
- **Timer**: prominent, counting up from 0:00.0 to 2:00.0, tenths precision
- **Judge panels**: three (or N) chunky bordered boxes at bottom — "JUDGE 1", "JUDGE 2", "JUDGE 3"
  - Default: lit, glowing, active
  - Buzzed: flashes, animated pixel X slams in, panel dims/reddens
- **CRT scanline overlay**: subtle CSS effect over entire screen

**Sounds** (play on audience screen only):
- Individual buzz: short buzzer sound effect
- Victory (timer reaches 2:00): triumphant chiptune fanfare
- Defeat (all 3 buzz): sad descending tone / sad trombone
- Last 10 seconds: accelerating tick sound, Countdown-style

**Animations**:
- Victory: screen shakes, pixel confetti/fireworks, "SURVIVOR!" text flashes
- Defeat: screen flashes red, "BUZZED OUT" drops in dramatically
- Last 10 seconds: timer goes red, pulsing glow, tension builds

### 3. Judge View (`judge.html`)

**Layout**: optimized for portrait mobile.

- **Header**: story code, judge number, story title
- **Timer**: counting up, visible but secondary
- **The Button**: one enormous red glossy 3D-ish CSS button, fills most of screen
  - Satisfying press animation: scale down + shadow collapse
  - After press: transforms to large animated X, cannot un-press
- **Other judges' status**: small indicators showing who else has buzzed (adds tension)
- **Between rounds**: shows "Waiting for next story..." with story title when queued

### 4. Controller View (`control.html`)

**Layout**: laptop/tablet optimized.

- **Session info**: story code (large, shareable), connected client count
- **Story queue**:
  - List of stories: title, truncated preview, reorder (drag), delete
  - "Add Story" form: title field + text area + add button
  - Current/next story indicator
- **Import/Export panel**:
  - Textarea showing session JSON (stories + config)
  - "Copy JSON" button, "Import JSON" button
  - Validate on import, show errors if invalid
- **Round controls**:
  - Select story / next story button
  - Start Round button
  - Play / Pause button
  - Speed toggle: 1x / 2x / 3x
  - Reset Round button
- **History log**: completed rounds — title, per-judge buzz times, outcome (victory/defeat), duration

## Auto-Scroll Mechanics

- Scroll is time-based, not position-broadcast
- Formula: `position = elapsed_time * speed_factor * base_pixels_per_second` (base rate: ~30px/sec, tuned during development)
- All clients compute scroll independently from synced timestamps
- Speed changes broadcast a `{speed, timestamp}` pair — clients recalculate from that point
- Pause broadcasts `{timestamp, elapsed}` — clients freeze. Resume broadcasts new reference point.
- No per-frame WebSocket traffic. Efficient, stays in sync.

## Timer

- Counts up from 0:00.0 to configured duration (default 2:00.0)
- Runs locally on each client from synced start time
- Display: tenths of a second precision
- Last 10 seconds: enters "tension mode" — red pulsing, accelerating ticks (Countdown-style)
- At limit: client fires victory, server confirms and broadcasts `round_end {victory}`

## Buzz Sequence

1. Judge presses button → client sends `buzz {judge_id}`
2. Server records buzz with server timestamp, adds to `current_round.buzzes`
3. Server broadcasts `buzzed {judge_id, time}` to all clients
4. Audience: judge panel animates to X, buzz sound plays
5. Other judges: see that judge's indicator change
6. Button disabled after press, server ignores duplicate buzzes from same judge
7. After final buzz (all judges buzzed) → server broadcasts `round_end {defeat}`

## Edge Cases

- **Judge double-tap**: button disabled client-side after press, server ignores duplicates
- **No judges connected when round starts**: controller gets warning, round proceeds
- **Controller disconnect mid-round**: round keeps running (timer is autonomous). Controller reconnects with same story code, gets full state, regains control.
- **Judge disconnect mid-round**: buzz state preserved on server. Reconnect = restored state.
- **Multiple tabs same role**: last connection wins, previous goes stale
- **Very short/long stories**: no auto-adjustment, scroll runs at set speed. Short stories finish fast, long ones get buzzed or win.

## Aesthetic

### Retro/Campy Game Show
- **Font**: Press Start 2P (Google Fonts) for headings, clean monospace for story text
- **Colors**: black background, neon green/amber/magenta accents. 80s terminal meets game show.
- **CRT effect**: CSS scanline overlay on audience view
- **Button**: glossy red, 3D CSS, chunky border, satisfying press feel
- **Judge panels**: chunky bordered boxes, neon glow, dramatic state changes
- **Overall vibe**: tongue-in-cheek, over-the-top, knowing camp

### Sound Design
- Buzz: short harsh buzzer
- Victory fanfare: triumphant chiptune jingle (2-3 seconds)
- Defeat: sad descending tone (2-3 seconds)
- Tension ticks: accelerating tick/beep in final 10 seconds
- All sounds: small .wav files in `./www/sounds/` or inline base64

## Future Provisions (not built now)

These are architected for but not implemented:

- **Global config UI**: `session.config` object exists, expose as settings panel later
- **Persistent storage**: `session.history` array is structured — swap for DB writes later
- **File-based story loading**: JSON import/export already handles batch loading — add file endpoint later
- **Flexible judge count**: judge count is a config variable, UI adapts to N judges
- **Author reveal**: story metadata can include optional `author` field — hidden during judging, revealable after
- **Multiple sessions**: server currently handles one session — add session routing later

## Out of Scope

- User accounts / authentication
- Multiple simultaneous sessions
- Persistent leaderboard / database
- Story file upload via browser
- Audience participation / voting
- Per-round timer configuration
